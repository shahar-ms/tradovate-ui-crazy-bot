# Tradovate Manual Trading Bot — Implementation Brief

Python manual-trading assistant for Tradovate (Chrome web app). This brief specifies a minimum viable bot that you can implement end-to-end and extend later.

Scope of this document:

1. A floating HUD showing live price and offering Buy / Sell / Cancel-All / Arm-toggle.
2. Price streamed from an on-screen region via OCR, with the improvements listed below.
3. Clicks to Tradovate delivered over the **Chrome DevTools Protocol** (CDP), NOT OS-level synthetic input. This is the single most important design decision — do not deviate from it. (See "Why CDP" below.)
4. A calibration tool that captures a screenshot, lets the operator draw rectangles / click points, and persists a `screen_map.json`. Calibrated targets: anchor region, price region, Buy button point, Sell button point, Cancel-All button point, position-size region, and a composite "position @ entry_price" region.

Out of scope for this iteration: strategy engine, risk management, multi-symbol, bar builders, replay, hotkey fallback. Architecture must leave room for those to be added.

---

## Non-negotiable lessons from prior work

These cost days of debugging in a previous iteration. Implement them correctly the first time.

### L1 — Use CDP for clicks, not SendInput/pyautogui

Chromium (M102+) applies trusted-event checks to incoming mouse events. A click dispatched via `SendInput` or `pyautogui.click()` from a Qt-hosted Python process **does reach Chrome at the OS level but is silently dropped by the renderer** — the cursor moves, no handler fires, no order is placed. This is not fixable by foreground tricks, `AttachThreadInput`, `SPI_SETFOREGROUNDLOCKTIMEOUT=0`, or subprocess isolation. It only worked from a bare script because the operator manually clicked Tradovate first to give it focus — the underlying click still goes through the same HID path.

CDP's `Input.dispatchMouseEvent` injects at the Blink render layer with the trusted flag set. Tradovate receives it as a normal user click. This is how Playwright / Puppeteer / DevTools all drive Chromium.

Cost: requires launching Chrome once with `--remote-debugging-port=9222`. This is a one-time shortcut edit.

Implement `PyAutoGUIClickDriver` as a fallback for non-Chrome clicks or diagnostic mode. The default click path must be CDP.

### L2 — DPI awareness before anything else imports

The screen-capture library (`mss`) reads **physical pixels**. If the process is not DPI-aware, `pyautogui.moveTo(201, 273)` lands at `(201 / scaling, 273 / scaling)`, which is the wrong pixel on a 125% or 150% display. Call `SetProcessDpiAwareness(PER_MONITOR_AWARE = 2)` at the very top of the entry script, before importing `pyautogui`, PySide6, or any other UI library.

### L3 — Single-instance guard

A `QLockFile` at `runtime/state/app.lock` prevents two copies of the bot running simultaneously. Two HUDs competing for the same calibration / cursor are a disaster. Auto-reclaim stale locks after 30 s so a crashed process doesn't permanently wedge the user out.

### L4 — Calibration persistence and drift detection

Calibration must persist across launches. On each startup, validate the saved `screen_map.json` by template-matching the anchor reference image against a fresh screen capture. If similarity < threshold (0.90), either the Tradovate window moved/resized — open the calibration dialog modally before the HUD appears. Do not silently click on stale coordinates.

The anchor region MUST be static text/chrome that never changes (DOM column headers like `BUY | BID | PRICE | ASK | SELL` work well). NEVER use prices, P/L values, timestamps, or chart content as the anchor.

### L5 — Position size is the source of truth for "am I in a trade?"

After a Buy/Sell click, do NOT try to detect the fill by diffing arbitrary pixel regions. OCR the broker's position-size cell (small integer: 0 = flat, ≥1 = in trade). Changes from 0 → N confirm the order filled. This is also how to detect externally-opened positions.

### L6 — Verified fill price via OCR of the entry-price region

The "position @ entry_price" composite reads the broker's own displayed entry price after a fill. Use this for PnL calculation, NOT the pre-click price observed by the bot — pre-click price ignores slippage. If OCR can't parse a fresh entry price within ~1.2 s, show `PnL: —` instead of an estimate.

## Data models (`app/models/common.py`)

```python
from pydantic import BaseModel

class Point(BaseModel):
    x: int
    y: int

class Region(BaseModel):
    left: int
    top: int
    width: int
    height: int

class ScreenMap(BaseModel):
    monitor_index: int
    screen_width: int
    screen_height: int
    anchor_region: Region
    anchor_reference_path: str     # relative to project root
    price_region: Region
    buy_point: Point
    sell_point: Point
    cancel_all_point: Point
    position_size_region: Region   # small integer cell
    # Composite: left=pos_size digit, middle='@', right=entry_price
    position_at_entry_region: Region
```

The composite `position_at_entry_region` is one rectangle covering all three sub-pieces. The OCR reader must split it (or OCR the whole strip and regex-parse `<int>\s*@\s*<price>`).

## Screen capture (`app/capture/screen_capture.py`)

- `mss` instance per thread (use `threading.local()`).
- `grab_region(region: Region) -> np.ndarray` returns BGR, not BGRA.
- Coordinates are physical pixels in virtual-desktop space (because DPI awareness is set).

## OCR reader (`app/capture/ocr_reader.py`, `preprocess.py`, `parser.py`)

Improvements learned in prior work:

1. **One preprocess recipe by default.** Previous code tried several (otsu, adaptive, inverted, ×2 scale). For broker UI (high-contrast white-on-dark numbers), a single Otsu threshold beats N recipes on speed with no accuracy loss. Keep the multi-recipe hook as an interface but default to `["otsu_threshold"]`.
2. **Return `(text, confidence)`**, not just text. Drop any result with confidence < 60 unless no alternative.
3. **Price parser** accepts optional comma thousand-separators, rejects outliers (e.g. below 0.01 or above 1e6 for MNQ).
4. **Integer parser for position size** accepts an empty/blank crop as 0 (the broker renders nothing when flat). Reject values outside `[-10, 10000]` as OCR noise. Treat small negatives (OCR confusing `-` with artifacts) as absolute value.

```python

```

## Price stream (`app/capture/price_stream.py`)

- Background thread. Captures `price_region` at ~15 fps.
- OCRs each frame. Emits a `Tick(price, confidence, ts_ms)` on change.
- Dedup: skip if OCR result == previous result (saves Tesseract calls on static screens). Use numpy pixel diff; if the crop is pixel-identical to last, skip OCR entirely.
- Health state: `ok` / `degraded` / `broken` based on rolling success rate.
- When health transitions to `broken`, the price poller keeps trying on a backoff (1 s → 2 s → 4 s → 10 s cap). Do not give up — Tradovate can briefly obscure the price region (tooltip, dropdown) and recover.

## Position watcher (`app/capture/position_watcher.py`)

- Same cadence and structure as price stream.
- Polls `position_size_region` with the integer parser.
- Emits on change, including `None → 0` on first successful read (treat blank crop as 0 = flat).

## Click driver (`app/execution/cdp_click_driver.py`) — PRIMARY

This is the critical component. Do not substitute pyautogui or SendInput.

```python
class CDPClickDriver:
    def __init__(self, host="127.0.0.1", port=9222, url_filter="tradovate"):
        ...

    def click_point(self, screen_x: int, screen_y: int) -> None:
        # 1) ensure WebSocket is connected to the Tradovate tab
        # 2) read window.screenX / window.screenY via Runtime.evaluate
        # 3) viewport_x = screen_x - screenX
        #    viewport_y = screen_y - screenY
        #    (divide by devicePixelRatio if != 1)
        # 4) Page.bringToFront
        # 5) dispatch:
        #    Input.dispatchMouseEvent {type:"mouseMoved", x,y, button:"none"}
        #    Input.dispatchMouseEvent {type:"mousePressed", x,y,
        #                              button:"left", buttons:1, clickCount:1}
        #    Input.dispatchMouseEvent {type:"mouseReleased", x,y,
        #                              button:"left", buttons:0, clickCount:1}
```

Startup sequence:

1. `GET http://127.0.0.1:9222/json` → list of targets.
2. Pick the target whose `type == "page"` and whose `url` contains `tradovate`.
3. Open a persistent WebSocket to its `webSocketDebuggerUrl`.
4. `Page.enable`.
5. Hold the connection for the lifetime of the bot. Reconnect on drop.

Error handling:
- If the port is unreachable, log a clear message: *"Chrome not running with `--remote-debugging-port=9222`. Relaunch Chrome with that flag."* and refuse to arm.
- If no tab matches the URL filter, list available tabs in the error so the operator can diagnose.

Thread safety: a single `threading.Lock` around send/recv. Only the executor calls it.

## Calibration (`app/calibration/`)

Flow (modal dialog shown on boot if `screen_map.json` missing or anchor similarity fails):

1. **Capture**: operator clicks "Grab screenshot". App grabs full monitor via `mss`, displays it in a QLabel.
2. **Draw / pick** each target in order. Each target has a type — `region` (drag rectangle) or `point` (single click):
   - `anchor` (region) — tight box on static DOM column header strip.
   - `price` (region) — just the price digits.
   - `buy` (point) — center of Buy Mkt button.
   - `sell` (point) — center of Sell Mkt button.
   - `cancel_all` (point) — center of Cancel-All button.
   - `position_size` (region) — the small integer cell (0 / 1 / …).
   - `position_at_entry` (region) — the composite strip `<size> @ <entry_price>`.
3. **Save**: on confirm, write `screen_map.json`, `anchor_reference.png` (cropped anchor), and a full `calibration_overlay.png` with every calibrated region/point drawn on top. The overlay is the operator's ground-truth sanity check.

Validator (`validator.py`):
- On boot, load `screen_map.json`, load `anchor_reference.png`, grab the anchor region from a fresh capture, compute `cv2.matchTemplate` similarity.
- If similarity ≥ 0.90, HUD opens normally.
- If < 0.90, open the calibration dialog modally. Cancel = exit cleanly.

## HUD (`app/ui/hud.py`)

Single `QWidget` window, frameless, always-on-top, no taskbar entry.

Window flags (PySide6):

```python
Qt.FramelessWindowHint | Qt.WindowStaysOnTopHint | Qt.Tool
```

Plus `Qt.WA_ShowWithoutActivating` on the widget so clicking the HUD doesn't steal focus from Chrome between ticks.

Layout:

```
+-----------------------------------------+
| BOT        [CAL ✓]   ⠿  [−]             |  20px  drag + minimize
|                                         |
|            26691.50                     |  36px  big price
|  health: ok          conf: 92           |  12px
|-----------------------------------------|
|  pos: flat                              |  12px  or  pos: LONG @ 26691.25
|  PnL: —                                 |  12px  hidden when flat
|-----------------------------------------|
| [ BUY ]  [ SELL ]  [ CANCEL ALL ]       |  32px
| [ ARM ON ]           [ Setup ]          |  28px  ARM toggles live clicks
+-----------------------------------------+
```

Behavior rules:

- `ARM` toggle: when OFF, Buy/Sell/Cancel buttons are disabled (or do-nothing). Only ARMED clicks reach the executor → CDP driver. Strategy auto-trading (future) also gated by ARM.
- `CAL ✓` / `CAL ✗` chip: green when screen_map loaded and anchor similarity current; red with tooltip *"Click Setup"* when calibration is stale/missing.
- All buttons `setFocusPolicy(Qt.NoFocus)` — clicking them must not steal focus from Chrome.
- HUD position persisted to `runtime/state/hud_pos.json` on move.
- Right-click menu: Reset position / Open Calibration / Open log folder / Exit app.
- A refresh timer (400 ms) pulls from `UiState` — never blocks.

   HUD.click → Controller.submit_manual("BUY")

## Entry point (`run.py`)

```python
# MUST be first. Before Qt / pyautogui / anything.
import sys
if sys.platform == "win32":
    import ctypes
    try:
        ctypes.windll.shcore.SetProcessDpiAwareness(2)   # PER_MONITOR_AWARE
    except Exception:
        ctypes.windll.user32.SetProcessDPIAware()

from app.ui.hud_app import boot
sys.exit(boot(sys.argv))
```

`hud_app.boot`:
1. Set up logging to `runtime/logs/session_<timestamp>.log`.
2. Create `QApplication`.
3. Acquire `QLockFile` at `runtime/state/app.lock`. If held, show a message box and exit.
4. Validate calibration; if invalid, open `CalibrationDialog` modally. Cancel = exit.
5. Create `AppSignals`, `UiState`, `UiController`.
6. Build `CDPClickDriver` — if it fails, log and let the HUD start anyway so the operator can recalibrate without a crash loop.
7. Create `FloatingHud`, restore saved position, show.
8. Start `PriceStream` + `PositionWatcher`.
9. `app.exec()`.

On quit: stop watchers, close CDP socket, release lock.

## Why CDP — short version for the skeptic

Tried in an earlier iteration, all failed to reliably click Tradovate's Buy Mkt button from a Qt host:

- `pyautogui.click()` (legacy `mouse_event`)
- `SendInput` with absolute MOVE + LEFTDOWN + LEFTUP, DPI-aware, on the virtual desktop
- Same, plus Alt-blip to reset foreground-lock
- Same, plus `AttachThreadInput` + `BringWindowToTop` + `SetForegroundWindow`
- Same, plus `SPI_SETFOREGROUNDLOCKTIMEOUT = 0`
- Same, plus minimize-then-restore of the Chrome window
- Subprocess isolation (dedicated short-lived Python process per click)

Cursor visibly lands on the button in all cases. Chrome does not register the click. The isolated script version worked ONLY because the operator had manually clicked Tradovate immediately before, giving the web page document focus. Recent Chromium builds apply trusted-event filtering on input whose metadata doesn't match a real HID device; the exact threshold is not documented.

CDP sidesteps the entire input stack. `Input.dispatchMouseEvent` fires events from inside Chrome, with `isTrusted = true`. It works every time. Playwright and Puppeteer use this. So do we.

## Out-of-scope hooks (don't build now, but don't paint into a corner)

- **Strategy engine**: a thread that consumes ticks from `PriceStream`, runs rules, emits Buy/Sell intents to the executor via a queue. Architectural requirement: make `Executor.buy()` and friends callable from any thread.
- **Ack reader**: after a click, poll `position_size_region` for ~1.2 s; when it changes, read `position_at_entry_region` for the verified fill price. Inject into `UiState.entry_price`.
- **Risk manager**: halts on N consecutive losing trades, session windows, max trades per day.
- **Replay mode**: feed `PriceStream` from a recorded CSV instead of live OCR.
- **Multiple brokers**: generalize `ScreenMap` → `BrokerProfile`.

Keep the executor / click-driver interface small (`buy`, `sell`, `cancel_all`, `click_point(x, y)`) so these can be added without touching it.
